Highlight #1 Version Control Systems

Version control is a system used to record changes to a set of files and their different versions over time. A good example of this is in different versions of software source code or webpage layout and graphics. A VCS is very wise in that it backups all changes made to files and stores them appropriately. If the developer ever messes up, they can easily revert or compare changes with their VCS. The local VCS is the most basic VCS. A centralized VCS allows multiple developers to collaborate. A distributed VCS, like Git, stores backups across multiple developer’s systems and is the most flexible VCS.

Highlight #2 History of Git

Git is a popular DVCS primarily used for course code management during software development, but also handy for sorting and backing up any form of files. Git was created in 2005 by Linux creator Linus Torvald, after former VCS Bitkeeper dissolved and a need for a faster, simpler, fully distributed VCS was needed for development with Linux. Without network access or a central server, all Git directories on all systems hold a full-fledged repository. It is aimed at faster processing, data integrity, and support for non-linear workflows. Git is distributed for free under the GNU general public license, like the Linux kernel.

Highlight #3 Git Basics

Git thinks about data differently than regular version control systems. VCS’s generally think of information kept as sets of files and changes made to these files over time. Git thinks of data more like snapshots of miniature filesystems. Every save is basically a snapshot of a map of a filesystem. For speed purposes, if a save is performed but no changes have been made, the snapshot will not resave but reference the last save instead. Git does not need to reach out to a server to provide a file history overview, instead nearly every operation of Git is done locally. This also allows for a lot of work to be done when the Git system is not connected to a network.

Highlight #4 Git Usage

Git files can reside in three main states: committed, modified, and staged. Committed files means that data is safely stored in a local database. Modified means that the file has been changed but not yet committed to the database. Staged means that a modified file is marked to be applied into the next commit snapshot. The three main sections of a Git project include the Git directory, the working tree, and the staging area. The Git directory stores metadata, the working tree is a single checkout of a single version of the project, and the staging area is where staged files are placed before committing.

Highlight #5 Branches

Almost all version control systems have a sort of branching support system, where a path diverges from the main line of development and continue to be built on without messing up the previous path. Branching can be an extremely exhaustive copy process for many VCS tools, but not for Git. Git branches are very lightweight and act instantaneously. With the help of this power, Git encourages workflows to merge or branch more often. Branches in Git are cheap to create and destroy since they are only 41 bytes to a file in size. This can take much longer in other VCS tools, especially with larger projects, but are instantaneous in Git.
